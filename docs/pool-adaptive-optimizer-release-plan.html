BlockDAG Pool Adaptive Optimizer Release Plan

The release design is guarded-adaptive and active-capable. The optimizer can mutate live pool knobs when runtime admin is available, but it uses long samples, small bounded steps, and damping cooldowns so it does not chase short-window luck or transient Pi I/O stalls.

Control Model

The optimizer runs three conceptual loops:

  1. Fast safety loop, 30 seconds: watch dashboard readiness, pool job health, MAC lane count, template age, submit readiness, ASIC hardware state, and host pressure. Transient warnings are recorded without ending the window; hard failures still stop or revert.
  2. Medium control loop, 15 minutes: adjust only one pool-local step per window in a bounded safe range: VarDiff target share interval, template TTL refresh, and max block-candidate job age.
  3. Slow reward loop, 30-120 minutes or settled sample minimum: promote settings only from source-truth accepted/blue/paid block conversion per active MAC miner-hour.

Objective

maximize:
  source_truth_blue_paid_blocks / active_mac_miner_hour

hard constraints:
  dashboard overall == ok
  can_mine && can_submit_blocks
  sustained pool_job_health_ok failure is absent
  stable active mac: lane count
  no current-job stale/invalidated miner condition
  no submit-disabled or transport-failed regression
  no settled source-truth conversion regression

secondary penalties:
  node reject rate by reason
  local stale/old-template/stale-parent reject rate
  stale-share ACK/reject rate
  p95 template fetch latency
  p95 submit latency
  backend switch churn
  clean-job broadcast churn
  Pi CPU/iowait pressure
  MAC lane work imbalance

Implemented Controller

ops/pool_adaptive_optimizer.py now implements the first release slice:

  • Apply mode requires --apply --yes and reachable local runtime admin endpoints; release systemd units use apply mode with damping.
  • MAC lane extraction from dashboard miner health so exposure and work share are based on mac: identities.
  • Safety scoring from dashboard health, share/block outcomes, stale reject reasons, template fetch latency, iowait, and ASIC lane health. High block rejects are a hard failure only when no accepted blocks are flowing; otherwise they are a corrective tuning signal.
  • Bounded recommendations for VarDiff target share interval, template TTL refresh, and block-candidate age.
  • JSONL evidence, state file, summary JSON, and HTML report for every run.
  • ops/install-dashboard.sh installs a low-priority periodic active timer by default when guard services are enabled; --no-adaptive-optimizer opts out.

OfficePool Baseline Finding

On 2026-05-28, OfficePool was mining with one X100 ASIC at mac:28:e2:97:1e:c0:b5. A 30 second live chain sample measured chain_block_count at roughly 1.48 blocks/s and chain_main_height at roughly 0.855 blocks/s, or about 1.17 seconds per main-chain height. Runtime admin was disabled in the previous live container, so release defaults now enable it on the localhost-bound metrics/admin port.

The live advisory samples produced useful hard-stop evidence. A 107 second sample measured 8 accepted block submissions, 1 old-template-age rejection, 51.52% share acceptance, 0.416 second average template fetch latency, and 53.57% host iowait; the controller held because the host was under too much I/O pressure. A second patched 45 second sample measured 5 accepted block submissions, 0 rejected block submissions, 88.24% share acceptance, 0.139 second average template fetch latency, and a pool_job_health_ok=0 abort; the controller held and recorded runtime_abort_pool_job_health_ok_0 plus pool_job_health_not_ok. A live active smoke after runtime-admin deployment showed accepted blocks still flowing while local stale block candidates were high, so the controller now treats that condition as a damped candidate-age reduction instead of freezing all tuning.

Steelman

The strongest objection is correct: dynamic tuning can be unsafe because rewards are sparse, delayed, and confounded by network/DAG conditions. One lucky or unlucky block can dominate a short window, and global knobs can degrade all miners at once. The answer is not free-running PID control; it is bounded safe exploration with a baseline arm, minimum evidence windows, hard stop rules, and source-truth settlement before promotion.

Next Release Slice

  1. Keep runtime admin enabled only on localhost/private management paths and keep host port publishing bound to localhost.
  2. Run active windows at 900 seconds, 30 second sample cadence, 30 minute timer cadence, and 30 minute damping cooldown.
  3. Apply only one bounded step per window; increase TTL/share interval under iowait pressure, reduce candidate-age on stale block-candidate evidence, and avoid changing multiple knobs at once.
  4. Add on-chain/blue/coinbase reconciliation into the optimizer score before unattended apply mode can be considered.
