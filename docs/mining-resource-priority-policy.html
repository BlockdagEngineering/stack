Mining Resource Priority Policy

This policy makes the mining critical path win local contention while keeping the system work-conserving. Idle CPU and I/O can still be used by dashboards and maintenance jobs, but when the pool, selected node, database, or RPC router need resources, they take priority.

Critical Path
paid blocks
Routing Model
active/passive
Timer
5 minutes
Avoided Risk
stale work

Why

BlockDAG mining runs on roughly one-second block cadence. Any avoidable local latency in template selection, Stratum job broadcast, submit handling, database writes, or peer propagation can turn solved work into stale, duplicate, overdue, or non-paying blocks.

Earlier live testing showed active/active template competition could reduce miner-normalized accepted block output by about 11-13%. The correct production shape is dual nodes for redundancy, one active canonical template lane, and fast failover only when the active lane degrades.

Policy

ClassExamplesRuntime Treatment
Active mining laneSelected bdag-miner-node-1 or bdag-miner-node-2Highest Docker CPU/blkio weight, highest process priority, best-effort I/O priority 0.
Pool and accountingasic-pool, pool-db, Postgres worker processesHigh Docker CPU/blkio weight, elevated process priority, best-effort I/O priority 0.
RPC pathrpc-failoverMedium-high Docker CPU/blkio weight, elevated process priority.
Standby nodeHealthy non-selected mining nodeProtected but lower than active lane, preserving fast failover without competing for the active template path.
Observability and supportGrafana, Prometheus, Loki, exporters, dashboard, release seeder, snapshots, browserLow CPU/IO weights or idle I/O so they yield under contention.

Implementation

ops/apply-mining-host-tuning.sh reads pool metrics to detect the selected backend. It then updates Docker CPU and block I/O weights and applies process nice/ionice values. It intentionally does not restart mining services or ASICs.

bdag-mining-host-tuning.timer reapplies the policy every five minutes and after boot. /etc/default/bdag-mining-host-tuning records BDAG_PROJECT_ROOT so the installed script can read the active release root.
