FastSnap Maintenance Resource Guard

Release-candidate guard for FastSnap seed exports on a live mining host. The export may intentionally drain one standby node, but it must not let archive work compete unbounded with the active mining path.

Mitigation added FastSnap CPU cap Live mining host

Failure Mode

ItemDetails
Observed triggerUSB FastSnap refresh on 2026-05-25.
Root causeBDAG_FASTSNAP_DOCKER_CPUS defaulted to empty, so Docker did not impose a CPU quota on the export or verify container.
Runtime symptomArchive finalization and verification reached roughly 1.1 to 1.5 CPU cores before manual capping. Pool remained online, but transient submit-error windows appeared during maintenance samples.
RiskOn smaller release hosts, uncapped maintenance work could starve the active backend, increase submit latency, or amplify stale/tip-overdue loss.

Permanent Guard

# FastSnap export finalization and verify are CPU-heavy. Keep the default
# bounded so maintenance cannot freely compete with the live mining backend.
DOCKER_CPUS="${BDAG_FASTSNAP_DOCKER_CPUS:-1.5}"

The value can still be overridden deliberately with BDAG_FASTSNAP_DOCKER_CPUS, but the release default is now bounded.

Acceptance Criteria

  • bash -n ops/build-fastsnap-seed.sh passes.
  • Release validation fails if the default CPU cap is removed without an explicit resource review.
  • Future maintenance reports include export duration, verify duration, CPU cap, selected backend, miner count, and submit-error deltas.
  • FastSnap export continues to drain only the standby backend and restores it before archive verification.
