# Redis Dash Upgrade Checkpoint Template

Use this template when creating a public-safe checkpoint for a redis-dash stack
upgrade. Put site-specific MAC addresses, LAN IPs, credentials, private paths,
bad peer addresses, and local firewall rules in private memory instead.

## Source Provenance

- `stack-redis`: target branch and full SHA
- `redis-dash`: target branch and full SHA
- `pool`: target branch and full SHA
- `blockdag-corechain`: target branch and full SHA
- memory repo: branch and full SHA

Record whether local tracked changes are part of the upgrade:

- mainnet network and production-named RPC defaults
- Docker Compose requiring explicit node RPC credentials
- source build contexts pointing at `redis-dash`
- any runbook, installer, or test changes

## Built Image Provenance

Record local image IDs in private memory. Public release docs should identify
image names, source commits, build logs, and release tags instead of one host's
ephemeral image IDs.

Required image names:

- `stack-node`
- `stack-pool`
- `stack-dashboard`

## Live State At Checkpoint

Record pass/fail evidence:

- `node` running
- `pool` running
- `postgres` running and healthy
- `dashboard` running and healthy
- optional route/guard services running when configured

Node health must include:

- `getTemplateHealth.reason_code=ok`
- `submit_ready=true`
- `mineable_now=true`
- `p2p_fresh_consensus_peer_count` at or above the configured floor
- stale consensus peer count not dominating the peer set
- peer lead inside tolerance

Dashboard and pool evidence must include:

- dashboard `/api/status` overall `ok`
- dashboard `/api/live/global` live, low data age, and moving stream
- producer plot continuous after redeploy
- pool ready miners match expected configured ASICs
- accepted shares and accepted block submissions increasing after restart

Observed ASICs must be recorded by MAC identity in private memory. IP addresses
are observations only.

## Runtime Configuration Checklist

The live `.env` and `node.conf` contain secrets and are gitignored. Public
checkpoints should record only the checks:

- `BDAG_NETWORK=mainnet`
- `.env` `NODE_RPC_USER` / `NODE_RPC_PASS` must match `node.conf` `rpcuser` /
  `rpcpass`
- `NODE_ARGS_APPEND` included `--miner`, the configured non-zero mining
  address, and `--maxinbound=8`
- `BOOTSTRAP_PEER_ADDRESSES` was narrowed to public mainnet multiaddrs with
  peer IDs
- `node.conf` kept public addpeer lines, cache tuning, and reduced log pressure

## Peerstore And Peer Mitigation

If peer hygiene affected the upgrade, record exact peer addresses only in
private memory. Public notes should describe the mitigation class:

- contaminated peerstore quarantined, not deleted
- stale peer blocked or removed from candidates
- inbound/outbound limits tuned to maintain the fresh consensus peer floor
- peer freshness validated before exposing Stratum

## What To Do Differently Next Time

1. Pull `codex-memory` and stack source before planning.
2. Build redis-dash, node, and pool images from source before stopping mining.
3. Preflight `.env` and `node.conf` for mainnet-only RPC auth and matching
   credentials.
4. Preflight public mainnet peer candidates and quarantine contaminated
   peerstores before exposing mining.
5. Use at least enough inbound capacity to maintain two or more fresh consensus
   peers.
6. Stop the pool only for the final replacement window.
7. Start and validate `node` before exposing `pool`.
8. Verify paid block-submission deltas, not just accepted shares.
9. Verify the dashboard layout and `/api/live/global` after redeploy.
10. Prune old images and source folders only after the human explicitly
    confirms the new stack.
