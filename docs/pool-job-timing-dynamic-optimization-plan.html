BlockDAG Pool Job Timing Dynamic Optimization Plan

Goal: maximize confirmed useful block production per miner-hour while minimizing stale/rejected work. The controller must treat paid block conversion as the primary metric and share rate as a telemetry/load knob, not as a way to increase the ASIC's actual probability of finding a network-valid block.

Verdict

Use conservative defaults plus bounded adaptive exploration. Fixed constants are safer than blind autotuning, but this stack has measurable signals for template age, submit latency, share rejection reason, node readiness, and accepted block outcomes. The right release-candidate design is a guarded controller that changes one pool-local parameter at a time, compares equal windows, and reverts immediately on stale/reject or source-truth regression.

Key Correction

Lowering pool share difficulty creates more submitted shares and therefore more feedback samples. It does not make the ASIC more likely to find a network-difficulty block. Too-low share difficulty can increase pool CPU, DB, network, and Stratum pressure, which can worsen stale work. The useful target is enough accepted shares to measure the miner and tune vardiff, with low rejected-share tails after clean job rotations.

Model

If the DAG tip changes at rate lambda_b and a candidate has effective age a when the node can validate/propagate it, then the expected newer-tip count is approximately Poisson:

N(a) ~ Poisson(lambda_b * a)
P(no newer tip) = exp(-lambda_b * a)

If BlockDAG acceptance tolerates up to k newer tips, the theoretical viability prior is:

P(lag <= k) = exp(-lambda_b*a) * sum(j=0..k, (lambda_b*a)^j / j!)

The release controller should not rely on this alone. It should fit an empirical q_blue(age, height_lag, backend_health, p2p_freshness) from source-truth block outcomes, then submit or reject locally based on expected value:

submit iff q_blue * block_value - stale_submit_cost > 0

Objective

maximize U(theta) =
  paid_blue_blocks_per_miner_hour
  - w1 * stale_block_rate
  - w2 * share_reject_rate
  - w3 * submit_p99
  - w4 * host_pressure
  - w5 * clean_job_churn

where theta includes template TTL, block-candidate job-age limit, same-template age limit, template lag limit, clean-job policy, and vardiff target share interval.

Harness

The harness lives at ops/pool_job_timing_optimizer.py. It defaults to observation only and writes summary.json, events.jsonl, and an HTML report. It can apply candidates only when run with --apply --yes, using local runtime-admin endpoints for template TTL, candidate age, and vardiff.

# Measurement-only smoke
python3 ops/pool_job_timing_optimizer.py --duration-seconds 900

# Controlled mutation, only after runtime admin is enabled and a human approves
python3 ops/pool_job_timing_optimizer.py --apply --yes --duration-seconds 1800 --warmup-seconds 300

Experiment Design

  1. Freeze ASIC speed, node topology, NAT/DHCP, and fastsync serving state.
  2. Run a 20-30 minute baseline window with no mutation.
  3. Change one knob at a time inside the safe prior range.
  4. Use 5 minute warmup after pool reconnect/restart, then 20-30 minute share/freshness windows.
  5. Use ABAB or rotating candidates for 4-8 hours before deciding on block-conversion improvements.
  6. Promote only settings that improve source-truth accepted/blue blocks per miner-hour without worsening stale/reject or host-pressure guardrails.

Abort Criteria

  • overall != ok, can_mine=false, or can_submit_blocks=false for more than 30-60s.
  • No accepted shares for 90-120s while the ASIC still reports hashing.
  • Current job stale/invalidated miners persist for more than 30s.
  • Template age exceeds 5s or submit-ready becomes false while mining continues.
  • Stale block/node reject slope doubles versus baseline.
  • Accepted block conversion drops by more than 15% after the minimum sample.
  • Host iowait remains above 25%, ASIC valid chips drop below 6, or hardware error ratio spikes.

Agent Review

Control/statistical review

The statistical agent recommended a bounded contextual bandit using Gamma-Poisson modeling for block production and Beta-Binomial modeling for stale/reject ratios. It explicitly rejected “more shares equals more blocks” and treated vardiff as a feedback/load knob.

SRE review

The SRE agent recommended measurement-only by default on the live Pi5. Runtime mutation should be pool-only, one knob at a time, with exact rollback and stop-loss criteria. It rejected unattended node, DB, Docker, ASIC, NAT/DHCP, or fastsync changes during timing experiments.

Steelman fixed settings

The strongest fixed-settings argument is that a 1s DAG is noisy and feedback loops can chase luck. The response is to preserve fixed settings as priors and limit exploration to narrow, reversible, source-truth-verified bands.

Release Plan

  1. Ship stale non-block share ACK grace and same-parent old-job block-candidate validation in the pool.
  2. Ship explicit vardiff timing env knobs and runtime-admin endpoint.
  3. Ship the observe-first harness and report format.
  4. Add source-truth confirmed blue/coinbase reconciliation into the harness scoring.
  5. Only then enable a guarded adaptive controller, defaulting to advisory mode until enough live evidence exists.
