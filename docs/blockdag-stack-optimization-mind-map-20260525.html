BlockDAG Stack Optimization Mind Map

Generated 2026-05-25 for the Pi5 mining stack release-candidate work. This is a planning and engineering-research document, not an implementation patch.

The stack has one business purpose: turn available miner work into valid accepted BlockDAG blocks with the least avoidable latency, drift, and downtime. The strongest optimization theme is not simply "add parallelism." On this host, the better path is to remove duplicate data transformations, make background work yield under pressure, and only spend mining-template work when miners can actually use it.

Central Thesis
do less wrong work

Most wasted CPU, RAM, and USB I/O comes from duplicate copies, repeated observers, template churn with no miners, and archive repackaging.

Next Epic
directory artifact sync

Make Fast Artifact Sync V2 publish and import verified directory chunks directly. Keep .bdsnap only as a compatibility bridge.

Primary Runtime Rule
sync and mining are modes

No miners means sync-only. Unsynced means catch-up priority. Mining template production should be lazy unless a miner can consume work.

SRE Rule
one truth sampler

Multiple repair loops repeatedly collecting status can become the outage. Consolidate expensive observation into one sampled state feed.

What The Blockchain Should Be Doing

The node should cheaply accept verified external block data, maintain canonical DAG ordering and EVM state, expose fresh templates only when mining can use them, serve trusted bootstrap state to other nodes, and avoid turning maintenance into extra write amplification. It needs to separate three modes that currently overlap too often:

  1. Catch-up mode: prioritize block/artifact import and peer throughput. Suppress template work, dashboard decoration, snapshot export, and repeated miner discovery.
  2. Mining mode: prioritize freshest template, lowest submit latency, low stale/duplicate share rate, and accepted paid blocks. Keep observer work cheap and route by measured quality.
  3. Serving mode: provide fast sync/artifact data to other nodes without pausing the active mining path or repackaging the same state repeatedly.
Unintuitive core idea:

On a Pi-class USB stack, more parallelism can be slower. Bounded parallel decode can help only when CPU is the bottleneck. When USB I/O, dirty writeback, memory pressure, or LSM compaction are the bottleneck, adding workers increases queueing and slows the one ordered writer that matters.

Detailed Findings

1. Fast Artifact Sync V2 Should Become The Primary Sync Primitive

The current code has the right ingredients: manifests, artifact roots, chunk descriptors, per-chunk verification, peer have ranges, resume ledger, and bounded parallel chunk downloads. The problem is that the Pi still pays for archive assembly and import. In cmd/fastsnap/main.go, V2 downloads chunks into a chunk directory, assembles a .bdsnap, verifies it, writes a sidecar manifest, and then the node imports the archive. In p2p/synch/fastartifact.go, the server still reads chunks from a monolithic snapshot archive.

The better long-term path is a verified directory artifact:

artifact-root/
  manifest.json
  chain/BdagChain/...
  chain/BdagChain/ancient/...
  evm/bdageth/chaindata/...
  evm/bdageth/ancient/...
  chunks-or-files indexed by path, offset, size, hash, source, class

For new nodes, bootstrap becomes: fetch manifest -> fetch/range-copy verified files or chunks -> fsync -> atomic rename into the datadir -> start live-tail FastSync from the manifest tip. There is no need to pretend every sync source must become one giant archive first.

2. Parallelism Must Respect The Ordered Writer

p2p/synch/fastsync.go can pre-decode and sanity-check trusted ranges in parallel, but accepted chain import still passes through BlockChain.ProcessBlock and an ordered state/database path. That serial section dominates once CPU decode is not the bottleneck. More workers are useful only until they saturate the single writer, memory, or USB. Beyond that point they create queues, cache churn, and latency spikes.

Use pressure-based adaptive controls: reduce artifact parallelism, global dashboard workers, miner API scans, snapshot rsync, and template prebuilds when /proc/pressure/io, iowait, dirty/writeback pages, or chain RPC latency indicate the host is waiting on storage.

3. The Template Path Has Avoidable Work

services/mining/newblocktemplate.go creates a timeout context but does not pass/check it through the expensive path. It also calculates state root before adding parents/transactions and then calculates it again after adding them. That first call may have side effects, so it must be verified before removal, but it is a clear audit target.

services/miner/gbtworker.go serializes transaction data and builds response structures per request. If the template sequence, payout address, pow type, and mutable timestamp bounds have not changed, the serialized payload can be cached. The high-value pattern is latest-wins: invalidate stale work aggressively rather than completing work for an old parent root.

4. No-Miner Sync-Only Must Be Operational, Not Cosmetic

The stack already records the invariant in AGENTS.md and runtime priority scripts: no miners means sync-only. The next improvement is to ensure the pool's own template loop obeys it. A host with zero connected/managed miners should not keep calling expensive template RPCs just to throw the result away. Health can use cheaper calls like getBlockCount or getTemplateHealth; expensive getBlockTemplate resumes only when a miner is connected or within a short recently-connected grace window.

5. Self-Healing Needs Less Self-Interference

The SRE critique is direct: too much self-healing can become an outage on a Pi 5. ops/dashboard.py, ops/watchdog.py, ops/p2p_guard.py, ops/stack_sentinel.py, and host/mining-appliance/bdag-runtime-priority all ask questions that can involve Docker, RPC, logs, and filesystem writes. Consolidating observation into one sampler lowers control-plane load and makes repair decisions more consistent.

Repair ownership also needs clear locking. A stopped node can be legitimately paused by the sync coordinator. A sentinel should respect that, but should still repair non-paused critical containers. Boot timers should be randomized and have grace periods to avoid stampeding after Pi resets.

6. Flash Storage Costs Are About Metadata Waves Too

The move to F2FS is directionally right for flash, but F2FS does not make metadata churn free. Snapshot hardlink trees, whole-file archive exports, recursive ownership changes, repeated JSON rewrites, Docker logs scraping, and repeated manifest scans all compete with chain DB writes. The target architecture should minimize metadata waves on the active USB partition and keep cold compatibility archives out of the hot path.

Steelman And Counterarguments

ProposalStrong CounterargumentAnswer
Directory artifact replaces .bdsnap as the main sync path. A single archive is portable, easy to copy, easy to verify, and operationally simple. Keep archive support as compatibility. Make new releases prefer directory artifacts when both sides support them. The manifest can preserve portability by defining a deterministic export command for offline handoff.
Use less parallelism on Pi. On better SSDs or x86 machines, more parallel download/decode may be faster. Make concurrency adaptive and host-profiled. The Pi default should be conservative; bigger hosts can expand when pressure signals stay low.
Lazy template production when no miners are present. The first miner to connect may wait for a template, reducing time-to-work. Build immediately on connection and keep a short warm window after disconnect. Zero miners overnight should not spend hours doing impossible revenue work.
Single status sampler. One sampler could become a single point of truth failure. Use per-field freshness, bypass paths for critical repair, and sentinel checks for sampler health. The current duplicate collectors are not independent enough to justify their cost.
Skip recursive startup ownership repair. Permissions can drift after manual restores or root-owned volume writes. Use targeted repair by ownership mismatch, stamp files, and an explicit full-repair switch. Full recursive repair should not run on every boot.
Return after first accepted gateway submit. Waiting for all peers gives more confidence and diagnostics. The mining critical path needs latency. Diagnostics can be collected asynchronously after the accepted response is returned.

Research Grounding

The online research does not change the local conclusions; it sharpens them:

  • Go's official diagnostics docs recommend profiling, tracing, and runtime stats to identify expensive code paths and scheduler/GC behavior. For this stack, pprof should be used on template building, FastSync import, and dashboard status collection before and after each optimization.
  • Linux PSI exists specifically to quantify CPU, memory, and I/O stalls so systems can shed or pause lower-priority work dynamically. That maps cleanly to dashboard global tabs, snapshot mirroring, miner discovery, and artifact parallelism.
  • F2FS is designed for flash and includes flash-aware allocation and background cleaning, but the kernel docs also expose GC and flush behavior. Reducing metadata churn and unnecessary copies still matters.
  • LSM-backed databases like LevelDB/Pebble/RocksDB pay write amplification through flushes and compaction. Geth's docs explicitly separate recent hot DB data from ancient/freezer data to reduce compaction cost. BlockDAG should treat artifact design and direct directory import as a way to avoid rebuilding the same LSM state from scratch.
  • Ethereum's snap-sync model demonstrates the same principle: start from a recent trusted state, then live-tail to head. It also shows that state sync must outpace chain growth. For BlockDAG, directory artifacts plus live FastSync ranges are the direct analogue.
  • Mining template protocols such as BIP 22/23 center on fresh work and long-poll/update semantics. The relevant lesson is not Bitcoin-specific: stale template work is wasted work, and miners should not wait behind obsolete template builds.
  • PostgreSQL WAL and async commit docs underline the durability/latency tradeoff. Pool accounting can batch non-critical bookkeeping, but accepted block records and payout-critical state need explicit durability policy.
  • Docker and systemd both provide resource controls, logging controls, and timer jitter. The stack should encode these as release defaults instead of redoing the same priority and timer behavior in many scripts.

Implementation Status - 2026-05-26

The concrete backlog above now has source-level implementation coverage in the release-candidate branches. The release default remains compatibility-safe: directory artifacts are preferred for Fast Artifact Sync V2, but .bdsnap archive fallback remains available for older peers and emergency recovery.

  • Measurement: ops/optimization_measurement.py records import/template/dashboard/resource evidence with host profile metadata.
  • Directory artifact V2: blockdag-corechain PR #112 defines directory manifests, verification, atomic install, file-chunk wire protocol, P2P serving, and fastsnap --dir-out/--install-dir. Stack entrypoints automatically serve a verified directory checkpoint after directory bootstrap.
  • No-miner sync-only: pool PR #121 defaults the pool template lane to idle with no authorized miners.
  • Startup and storage: entrypoint ownership repair is stamped/targeted by default, and directory artifact bootstrap avoids archive assembly/import when capable peers exist.
  • Control plane load: shared status sampling, append-only histories, adaptive worker counts, and pressure-aware background maintenance are wired into the stack source.
  • Repair safety: timers use jitter/boot grace and shared locks; deploy updates now include a post-restart health gate with rollback.
  • Mining latency: template serialization cache, no-miner lazy templates, submit dedupe, and candidate-only submit fanout are implemented in the pool/corechain source branches.

Measurement Plan

Before changing code, run a baseline in three states: no miners syncing, no miners synced, and miners connected mining. Store results with commit SHA, image tag, filesystem, mount options, USB device, and dashboard URL. For every optimization, compare against the same scenario.

  • Sync/import: blocks/sec, p50/p95/p99 ProcessBlock, FastSync range size, orphan/duplicate counts, mempool regain frequency, DB read/write bytes, compaction/write stall metrics where exposed.
  • Artifact: manifest time, chunk fetch p95, retries, bytes in flight, RSS, GC, archive assembly time, verification time, import time, total bytes read/written.
  • Template: getBlockTemplate p95/p99, template build phase timers, pending tx count, state-root time, serialized response time, stale template drops.
  • Submit: submit p95/p99, first accepted latency, duplicate candidate rate, stale/overdue/tip mismatch, node rejection reasons.
  • Control plane: dashboard status latency, Docker CLI calls/minute, log bytes scanned, JSON writes/minute, timer overlap after boot, watcher loop time.
  • Host pressure: iowait, /proc/pressure/io, dirty/writeback pages, load average, throttling/thermal state, USB errors/reset evidence.

Final Recommendation

Build the next release work around a single performance principle: do not move, decode, verify, rewrite, or observe data unless that work directly advances sync, keeps mining safe, or improves accepted block output. The clean future shape is:

Directory artifact bootstrap -> live FastSync tail -> lazy mining templates -> latency-scored submit/router -> single cheap status sampler -> pressure-aware maintenance

That path is more elegant and more practical for the Pi5 than adding blanket parallelism. It also makes the system easier for future agents to reason about: every background activity has an explicit budget and a reason to yield when sync or mining is at stake.
